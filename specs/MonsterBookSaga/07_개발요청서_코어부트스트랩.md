# 개발요청서 #1 — 순수 Lua 코어 부트스트랩 (T1~T3)

> 이 문서는 [[올림푸스-기획요청서-작성요령]] 템플릿을 따른 **실행 요청서**다.
> `00~06`이 프로젝트 정본 기획이라면, 이 문서는 그중 **"지금 이번 런에서 굴릴 좁은 범위"**를 콕 집어 올림푸스가 바로 분석·개발·검증하도록 지시한다.
> 대상 범위 = `05_개발태스크.md`의 **T1·T2·T3** (순수 Lua 코어의 토대). T4~는 이 요청서가 green 된 뒤 다음 요청서로 분리한다.

## 목적
메이플 몬스터북 사가의 **MSW 비의존 순수 Lua 게임 코어**를 시작할 수 있는 최소 골격을 세운다.
구체적으로 ① 코어 폴더 구조 + 표준 Lua로 단독 실행되는 경량 테스트 러너 + `verify.sh` 게이트, ② 결정론 시드 RNG/셔플 유틸, ③ 콘텐츠(03) 데이터 로더 + 스키마·참조 무결성 검증. 이 세 가지가 서면 이후 모든 규칙·전투 코드가 올릴 **검증 심판(verify.sh)** 과 **결정론 기반**을 갖춘다. 코어는 MSW·luarocks 등 외부 의존 없이 표준 Lua 인터프리터(CI/올림푸스)에서 단독 실행되어야 한다(→ `04_기술아키텍처_MSW.md` §4.1·4.4).

## 대상
- repo: **greenfield 신규 repo** `monsterbook-saga-core` (예상 경로 `~/IdeaProjects/monsterbook-saga-core`). 없으면 새로 생성(git init).
  - ⚠ 이건 **순수 Lua 코어 전용 repo**이지 MSW 에디터 프로젝트가 아니다. MSW 바인딩(T11~)은 별도.
  - 실행 시 `TARGET_REPO`로 지정하거나, spec 폴더가 이 프로젝트 하나만 되도록 정리(`Mindboard/` 등 치우고 `bash reset.sh`) — 알파벳순 첫 폴더 오선택 방지(작성요령 §4).
- 수정/생성 파일(예상): `core/` 하위 소스, `spec/` 또는 `test/` 하위 테스트, `verify.sh`, 최소 `README.md`. 이 범위 밖 파일은 건드리지 않는다.

## 요구사항 (동작 명세)

### R1 — 프로젝트 골격 + 테스트 하네스 (T1)
1. 코어 소스 폴더(`core/`)와 테스트 폴더 구조를 만든다.
2. **경량 Lua 테스트 러너를 저장소에 벤더링**(외부 luarocks 설치 불필요). 표준 `lua`(또는 `lua5.4`) 하나로 전 테스트 실행 가능해야 한다.
3. `verify.sh`가 그 러너를 실행하고 **테스트 결과로 종료코드**를 낸다(전부 통과=0, 하나라도 실패=비0). 올림푸스 범용 verify(lint/test/build for npm)가 아니라 **이 Lua 러너 호출로 대체**한다(04.6).
4. **입력→출력 예시**: `bash verify.sh` 실행 → 더미 테스트 1개 통과 출력 → 종료코드 `0`.

### R2 — 시드 RNG + 유틸 (T2)
1. 시드로 초기화하는 결정론 RNG(같은 시드 → 같은 난수 시퀀스). 전투 내 모든 무작위의 유일 원천이 될 인터페이스(예: `Rng.new(seed)`, `rng:nextInt(lo, hi)`).
2. 결정론 셔플(피셔-예이츠 등, RNG만 사용)과 안전 정수 연산 유틸.
3. **입력→출력 예시**:
   - `Rng.new(42)`로 뽑은 정수 시퀀스 == 다시 `Rng.new(42)`로 뽑은 시퀀스 (재현).
   - 같은 시드로 같은 배열을 셔플 → 항상 동일 순열. 다른 시드 → (거의) 다른 순열.

### R3 — 데이터 로더 + 스키마 검증 (T3)
1. `03_콘텐츠_데이터스펙.md`의 논리 스키마(Card / Monster / Job / Encounter / Map / Balance)를 **Lua 테이블 모듈**로 로드한다(04.3 권장 포맷).
2. **필수 필드 검증** + **참조 무결성 검증**: 카드의 `monsterId`가 존재하는 몬스터를 가리키는지, `cardId`/`jobTag` 등 상호 참조가 유효한지 확인. 깨진 참조·필수필드 누락은 **어느 항목의 어떤 필드인지 명확한 에러 메시지**로 실패시킨다(조용한 통과 금지).
3. 최소 샘플 데이터셋(03의 예시 엔트리 `warr_taunt`·`mage_fireball`·`mon_snail`·`snail` 등 소수)을 포함해 로더가 실제로 도는 걸 보인다.
4. **입력→출력 예시**:
   - 유효 샘플셋 로드 → 성공(에러 없음), 카드/몬스터 개수 등 로드 결과 반환.
   - `monsterId="does_not_exist"`인 카드를 넣은 셋 로드 → `"card 'X' references unknown monsterId 'does_not_exist'"` 류의 에러로 실패.

## ⚠ 반드시 사람에게 확정받아야 하는 값 (가정 금지 — 진짜 블로커)
- **없음 — 모두 합리적 가정으로 진행 가능.**
  - repo 이름/경로, 테스트 러너 종류, 파일 배치, RNG 알고리즘, 로더 API 형태는 모두 코어 내부 결정이라 임의 선택해도 무방(이후 리네임·리팩터 가능). 데이터 스키마는 `03`에 이미 정의됨.
  - 단 아래 §전제(가정 금지가 아니라 **명시 제약**)는 지켜야 한다 → 요구사항에 이미 못 박음: (a) 외부 luarocks 의존 없이 표준 Lua로 실행, (b) MSW API 미사용, (c) verify는 Lua 러너 기반.

## 수용 기준 (그라운드 트루스)
- `bash verify.sh` 종료코드 `0` (전 테스트 통과) — 이게 이 프로젝트의 심판 게이트.
- **T1 AC**: 더미 테스트 1개 green, `verify.sh` 종료코드 0.
- **T2 AC**: 같은 시드 → 같은 난수 시퀀스/셔플 재현하는 단위테스트 통과.
- **T3 AC**: 샘플 데이터셋 로드 성공 테스트 + 깨진 참조가 명확한 에러로 실패하는 테스트, 둘 다 통과.
- 표준 Lua 인터프리터 하나로 외부 패키지 설치 없이 전 테스트 실행 가능(환경 재현성).
- (신규 repo이므로 기존 테스트 회귀 항목 없음.)

- 효과 DSL 해석기(T4), 전투 상태/턴 엔진(T5~T7), 직업 레이어(T8), 런/맵/보상/메타(T9), 밸런스 시뮬(T10) — **다음 요청서로 분리**. 이번엔 만들지 않는다.

- 밸런스 튜닝, 성능 최적화.

## 실행 (참고)
```bash
cd ~/IdeaProjects/olympus
bash reset.sh
cp ~/sj-wiki/specs/MonsterBookSaga/*.md ~/IdeaProjects/olympus/spec/MonsterBookSaga/   # 정본 복사(_사용법 참조)
TARGET_REPO=~/IdeaProjects/monsterbook-saga-core \
REMOTE=1 POLL_TIMEOUT_SEC=1800 \
VERIFY_CMD="bash verify.sh" \
DEPLOY_ON_DONE=0 \
  bash run.sh
```
- 이 코어 repo는 greenfield라 처음엔 `verify.sh`가 없다 → 올림푸스가 R1에서 만든 뒤 게이트로 쓴다. 초기 스캐폴딩 동안 verify가 없다고 멈추면 `VERIFY_CMD`를 첫 통과 후 지정하거나, run 초반 부트스트랩만 수동으로 한 번 커밋해두는 방식 고려.
- `DEPLOY_ON_DONE=0` = 배포 스텝 skip(코어 라이브러리라 배포 없음).
