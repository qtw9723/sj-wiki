# 04. 기술 아키텍처 (MSW) + 자동검증

> ⚠ MSW 실제 개발 방식은 공식 문서 개념 수준까지만 확인됨. 아래 **§4.5 확인 필요 항목**을 손검증한 뒤 세부를 확정한다. 단 **§4.1 코어/표현 분리 원칙**은 확인 결과와 무관하게 유지한다.

## 4.1 핵심: 코어(순수 Lua) ↔ MSW 바인딩 분리
게임을 두 계층으로 나눈다.
- **게임 코어 (Pure Lua, MSW 의존 0)**: 전투 엔진, 카드/효과 DSL 해석기, 더미/드로우, 런 진행, 시드 RNG, 데이터 로더. **부작용 없음**: 입력(상태+행동+시드) → 출력(다음 상태). 순수 함수적.
- **MSW 바인딩 계층**: 입력(터치/클릭)→코어 액션 변환, 코어 상태→화면 렌더, 저장/불러오기, 에셋(스프라이트/사운드) 연결, UI.

효과:
1. 코어를 **표준 Lua 인터프리터로 CI에서 단독 실행** → 헤드리스 시뮬·자동검증 가능(사람 검증시간 절감).
2. MSW API가 바뀌거나 손검증으로 방식이 달라져도 **규칙 코드는 안 건드림**.
3. 결정론 보장 지점이 코어 한 곳으로 모임.

```
[MSW UI/입력/에셋/저장]  <->  [바인딩 어댑터]  <->  [순수 Lua 코어 (규칙·전투·데이터)]
                                                        ^ CI에서 이 부분만 단독 시뮬/테스트
```

## 4.2 MSW 개념 매핑 (공식 문서 기준, 개념 확인됨)
- MSW는 **서버-클라이언트 권위 모델** + **Entity/Component/Property** 구조, **프로퍼티 동기화**로 상태 일치, **자동호출 이벤트 함수**(라이프사이클) 사용. (출처: MSW Creator Center)
- 1인용이므로 권위 로직은 최소화 가능하나, **영속 저장은 서버 측**일 가능성 큼(→ §4.5).
- 잠정 매핑:
  - 전투/런 상태 = 코어가 소유하는 순수 데이터(프로퍼티로 노출은 렌더용 최소).
  - 카드/필드/적 = UI 엔티티(컴포넌트)로 표현, 코어 상태를 구독해 갱신.
  - 입력 = 이벤트 함수에서 코어 액션 호출.

## 4.3 데이터 로딩
- 콘텐츠(03)는 **Lua 테이블 모듈**로 두고 코어가 `require`/로드(권장, 표준 Lua에서도 로드 가능해 CI 친화). MSW가 JSON 로딩을 강제하면 어댑터에서 파싱. (→ §4.5 확인)

## 4.4 자동검증 전략 (verify.sh 게이트)
올림푸스 심판 게이트를 이 프로젝트용으로 정의한다. **코어가 순수 Lua라 MSW 없이 실행**하는 게 전제.
- **단위 테스트**(busted 또는 경량 Lua 테스트 러너): 피해/방어 계산, DSL 각 키워드, 드로우/셔플, 상태이상 스택.
- **불변식 시뮬**: 랜덤 시드 다수 × 스크립트 정책으로 전투 자동 진행 → 매 스텝 assert:
  - `hp>=0`, `block>=0`, 에너지 보존, **카드 집합 보존**(유실/복제 없음), 유한 턴 종료(≤max_turns).
- **결정론 리플레이**: 같은 (시드+행동열) → 동일 최종 상태 해시.
- **밸런스 회귀**: 기본 정책 봇의 1막 승률이 [40%,70%] 범위(⚙️) 밖이면 경고(실패 아님, 지표).
- `verify.sh`는 위 러너를 실행하고 종료코드로 green/red 판정. (표준 게이트의 lint/test/build을 Lua 코어 테스트로 대체)

## 4.5 ⚠ MSW 손검증 필요 항목 (개발 착수 전 확인 → 세부 확정)
1. **Lua 런타임**: MSW의 Lua 버전/샌드박스. **표준 Lua 인터프리터로 코어를 단독 실행 가능한가**(CI 시뮬 전제). 불가하면 코어를 MSW 의존 최소 API만 쓰도록 좁히거나, 로직을 순수 함수로 유지하되 테스트 하네스를 MSW 헤드리스로 대체.
2. **스크립트 라이프사이클/이벤트 함수** 정확한 이름·시그니처(초기화/매틱/입력 콜백).
3. **UI 시스템**: 카드/손패/필드/의도 아이콘을 그리는 방법(내장 UI 컴포넌트·좌표계·터치 입력).
4. **영속 저장**: 유저별 도감/해금/통계 저장·불러오기 API(서버 스토리지/DB). 1인용 세이브 모델.
5. **에셋 사용**: 콘테스트 제공 메이플 몬스터 스프라이트·직업 스킬 이펙트의 목록·참조 방법(`spriteRef` 채우기).
6. **1인용 구성**: 멀티 지향 MSW에서 싱글플레이 월드 구성이 자연스러운지(서버/클라 분리 최소화 가능 여부).
7. **배포/제출 형식**: 콘테스트 제출이 MSW 월드 게시 형태인지(빌드 산출물 정의 → 올림푸스 verify의 "build" 의미 확정).

## 4.6 올림푸스 연동 메모
- MSW는 일반 repo가 아니라 자체 에디터 프로젝트 → 올림푸스 **greenfield 자동 repo 생성(git init+범용 verify.sh)이 그대로 안 맞을 수 있음**. `templates/verify.sh`를 **Lua 코어 테스트 러너 호출**로 교체한 커스텀 게이트를 이 프로젝트에 둔다.
- 코어/바인딩 분리 덕에 **올림푸스는 코어(순수 Lua)를 주 개발·검증 대상**으로 삼고, MSW 바인딩·UI·에셋은 손검증 후 사람이 붙이는 반자동 경로로 둘 수 있음(위험 격리).
