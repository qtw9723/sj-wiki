## 한 줄 정의
📄 교회에서 쓸 **일정 공유 캘린더** — 여러 관리자가 일정을 등록하고, 사용자는 자신이 속한 **카테고리(팀/그룹)** 의 일정만 골라 본다. 로그인·권한 관리 필수, **1차는 대학부** 대상, 추후 **모바일 앱·ERP** 확장 대비.

> [!note] v1.1 개정 (2026-06-16, 사용자 피드백 반영)
> 기획 3종 문서 개정: ① 목차(실제 페이지번호) 추가 ② 배경=여러 카톡방 분산·공지 누락/지연 ③ 그룹 예시=목양기관·봉사기관·교역자 ④ 역할 시나리오=대학부 사역자/교역자 ⑤ **권한은 카테고리별 부여**(A=일반, B=관리자 가능) ⑥ **영혼관리시스템 연동** 위해 식별 기본정보 수집→매핑 ⑦ 로드맵 순서=캘린더→모바일→알림→ERP(인력·재정/회계 제외) ⑧ 스택 "협의 후 변경 가능" 명시 ⑨ 고령 접근성 항목 삭제 ⑩ **1차 대학부**→가용성 따라 교회 확장.

## 문제 / 배경
- 📄 교회 일정이 단체 메신저·주보·구두 등 분산 채널로 공유돼 누락·중복·장소 충돌 발생. (요구사항 기반 정리)
- 🧠 소속에 따라 봐야 할 일정이 다른데 모두 섞여 전달 → 정작 본인 관련 일정을 찾기 어려움 (배경 추론).

## 핵심 요구사항 (사용자 명시) 📄
1. 핵심 기능은 **일정 공유용 캘린더**.
2. **여러 관리자**가 직접 일정을 등록, **여러 사용자**가 조회.
3. 사용자·관리자는 **카테고리(팀/그룹)별**로 일정을 등록하고, 해당 카테고리 소속자에게만 그 일정이 보인다.
4. **모든 사용자 로그인 필수**, **권한 관리 화면** 필요.
5. 추후 **모바일 앱**(일정 등록·변경 시 알림) 및 **ERP(인력·재정/회계 제외)** 확장 → **확장성 있게** 설계.
6. 스택: **웹 표준(React + Node/Spring)**(협의 후 변경 가능), 규모: **소규모(~수백 명)**.
7. **1차 적용 대상은 대학부**, 검증 후 가용성에 따라 교회 전체로 확장.
8. 추후 **영혼관리시스템 연동**을 위해 개인 식별 기본정보를 수집·매핑.

- 📄 **같은 교회 개발팀의 신규 프로젝트**로, [[교회-백엔드-회의-2026-06-12]]와는 **다른 구성원들과** 진행. (확정 2026-06-16)
- 🧠 설계 관점에서 권한 모델·데이터 모델·API·확장 전략을 직접 설계 (백/프 협업 문서로 활용).

## 권한 · 카테고리 설계 (핵심)
- 📄(확정) **권한은 카테고리별로 부여**된다. 같은 사용자가 A 카테고리에서는 일반 사용자, B 카테고리에서는 관리자일 수 있다.
- 📄(확정) **권한 부여 주체**: 전체 관리자 = 모든 카테고리에 모든 권한 부여 가능 / 카테고리 관리자 = 자신이 관리하는 카테고리의 일반 사용자 등록·관리.
- 🧠 설계안: ① **카테고리 스코프 역할**(기본 축, category_membership.role_in_category=MEMBER/ADMIN) + ② **글로벌 역할**(ADMIN/SUPER_ADMIN, 시스템 전반).
- 🧠 **가시성 규칙**: 일반 사용자 = 본인 소속 카테고리 + 전체 공개 / 카테고리 관리자 = 담당 카테고리(편집) + 본인 소속 / 전체 관리자 = 전부. **서버측 강제 필터링**(프론트 신뢰 X).
- 🧠 **데이터 모델 핵심**: user(+식별 기본정보: name·birth_date·phone·soul_mgmt_ref) · role · permission · category(self-ref) · category_membership(다대다) · event · event_category(다대다) · event_occurrence. 시각 UTC 저장→KST 표시.

## 영혼관리시스템 연동 (향후) 📄(확정)
- 회원가입·등록 시 개인을 식별할 수 있는 **기본정보(이름·생년월일·연락처 등)를 수집**.
- 당장은 캘린더 내부에서만 사용하고, 추후 **영혼관리시스템이 준비되면 식별정보 기준으로 매핑**하여 연동(soul_mgmt_ref에 외부 키).
- 🧠 개인정보 최소 수집·접근통제·동의 필요.

## 기술 스택
- 📄 프론트: **React** (+ TypeScript, 반응형) / 백엔드: **Node 또는 Spring** 중 택1.
- 🧠 구체화 권장: 백엔드 **NestJS(TS) 또는 Spring Boot(Java)** — 사내 역량 기준 택1, **PostgreSQL**, JWT+Refresh 인증, OpenAPI 문서, 컨테이너 배포.
- ⚠️ 🧠 [[공통-기술스택]](개인: React19+Vite+Tailwind+**Supabase**)과 다름 — 이 프로젝트는 협업·ERP 확장 때문에 **자체 API 서버(Nest/Spring)+PostgreSQL** 노선. Supabase로 갈지(빠른 1차) 자체 서버로 갈지는 팀 합의 사항.

## 확장 로드맵
- 📄(확정) 순서: **Phase1 공유 캘린더 → Phase2 모바일 앱(일정 등록·변경 시 푸시 알림) → Phase2.5 알림·편의 → Phase3 ERP(인력·재정/회계 제외: 자산·자원 예약·출석/통계·문서)**.
- 📄(확정) 모바일을 알림 고도화보다 **먼저** 진행(푸시 사용 경험 우선).
- 🧠 확장 핵심 수칙: 1차부터 **인증·사용자·권한·카테고리를 공통 모듈로 분리** → 모바일·ERP는 그 위에 모듈 추가(캘린더 재작성 X), API 버저닝(/v1).
