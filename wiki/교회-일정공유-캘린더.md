## 한 줄 정의
📄 교회에서 쓸 **일정 공유 캘린더** — 여러 관리자가 일정을 등록하고, 사용자는 자신이 속한 **카테고리(팀/그룹)** 의 일정만 골라 본다. 로그인·권한 관리 필수, 추후 **ERP(인력관리 제외)·모바일** 확장 대비.

## 문제 / 배경
- 📄 교회 일정이 단체 메신저·주보·구두 등 분산 채널로 공유돼 누락·중복·장소 충돌 발생. (요구사항 기반 정리)
- 🧠 소속에 따라 봐야 할 일정이 다른데 모두 섞여 전달 → 정작 본인 관련 일정을 찾기 어려움 (배경 추론).

## 핵심 요구사항 (사용자 명시) 📄
1. 핵심 기능은 **일정 공유용 캘린더**.
2. **여러 관리자**가 직접 일정을 등록, **여러 사용자**가 조회.
3. 사용자·관리자는 **카테고리(팀/그룹)별**로 일정을 등록하고, 해당 카테고리 소속자에게만 그 일정이 보인다.
4. **모든 사용자 로그인 필수**, **권한 관리 화면** 필요.
5. 추후 **ERP(인력관리 제외)** 및 **모바일 앱** 확장 → **확장성 있게** 설계.
6. 스택: **웹 표준(React + Node/Spring)**, 규모: **소규모(~수백 명)**.

## 내 역할

## 권한 · 카테고리 설계 (핵심) 🧠
> 이 절은 raw 요구(로그인·권한·카테고리 노출)를 바탕으로 한 **Claude의 설계안**이다.

- **2단계 권한 모델**: ① **글로벌 역할(Role)** = 무엇을 할 수 있는가 (Member / Category Admin / Admin / Super Admin), ② **카테고리 스코프 역할** = 어디에 적용되는가 (category_membership.role_in_category).
- **가시성 규칙**: 일반 사용자 = 본인 소속 카테고리 + 전체 공개 / 카테고리 관리자 = 담당 카테고리(편집) + 본인 소속 / 전체 관리자 = 전부.
- **서버측 강제**: 일정 목록 조회 시 백엔드가 항상 접근 가능한 카테고리로 필터링 (프론트 필터 신뢰 X).
- **데이터 모델 핵심**: user · role · permission · role_permission · user_role · category(self-ref parent_id) · category_membership(다대다) · event · event_category(다대다) · event_occurrence(반복 전개/예외). 시각은 UTC 저장 → KST 표시.

## 기술 스택
- 📄 프론트: **React** (+ TypeScript, 반응형) / 백엔드: **Node 또는 Spring** 중 택1.
- 🧠 구체화 권장: 백엔드 **NestJS(TS) 또는 Spring Boot(Java)** — 사내 역량 기준 택1, **PostgreSQL**, JWT+Refresh 인증, OpenAPI 문서, 컨테이너 배포.
- ⚠️ 🧠 [[공통-기술스택]](개인: React19+Vite+Tailwind+**Supabase**)과 다름 — 이 프로젝트는 협업·ERP 확장 때문에 **자체 API 서버(Nest/Spring)+PostgreSQL** 노선. Supabase로 갈지(빠른 1차) 자체 서버로 갈지는 팀 합의 사항.

## 확장 로드맵
- 📄 확장 대상: **ERP(인력관리 제외)** + **모바일 앱**.
- 🧠 단계안: Phase1 캘린더(MVP) → Phase1.5 알림·편의 → Phase2 ERP(재정/회계·자원예약·출석/통계·문서) → Phase3 모바일(동일 API 재사용·푸시).
- 🧠 확장 핵심 수칙: 1차부터 **인증·사용자·권한·카테고리를 공통 모듈로 분리** → ERP·모바일은 그 위에 모듈 추가(캘린더 재작성 X), API 버저닝(/v1).
