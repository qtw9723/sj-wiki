## 🔥 빅 이슈

### ChatGPT Work — "몇 시간이고 프로젝트를 붙잡는 행동 에이전트" 📄
*2026-07-09 · OpenAI Blog*

**ChatGPT Work**는 **앱과 파일을 가로질러 행동(action)하고, 필요하면 몇 시간이고 한 프로젝트에 머물며, 목표를 완성된 결과물로 바꾸는 에이전트**. "가장 야심 찬 일의 파트너"로 포지셔닝.

> 🧠 **W26 CUGA·W27 MIT 에이전트 Q&A의 제품화 지점**: "에이전트가 실무를 대신한다"가 연구·데모를 넘어 **오피스 생산성 제품**으로. 내 [[에이전트-자동화-도구]] 정리(workflows·/goal·routines)와 같은 계열이고, [[mailer|CS SmartHub]]처럼 "여러 도구를 가로질러 하나의 결과물을 만드는" 내 자동화 방향과 정확히 겹친다. CS팀 반복 업무를 "몇 시간짜리 완성 작업"으로 위임하는 시나리오를 상상해볼 지점.

### GPT-5.6 — 프런티어 모델 + M365 Copilot 기본 채택 📄
*2026-07-09 · OpenAI Blog*

**GPT-5.6**: "토큰당 더 많은 지능, 달러당 더 강한 성능, 필요 시 더 큰 능력". 같은 날 **Microsoft 365 Copilot의 선호(preferred) 모델**로 채택 — Word·Excel·PowerPoint·Chat·Cowork 전반을 구동.

> 🧠 **[[CogInsight-Generator]] 모델 업그레이드 후보**: 현재 gpt-4o로 6단계 생성 파이프라인을 돌리는데, "성능/달러" 개선이 셀링포인트인 GPT-5.6은 **비용·품질 재검토 대상**(W27 "최신·최대 ≠ 최적" 기조와 함께 저울질). M365 Copilot 기본 채택은 내가 쓰는 [[Teams-Gmail-캘린더-Gemini-연동|M365·Cowork]] 환경에도 직접 영향.

### 코딩 벤치마크의 신호와 잡음 — SWE-Bench Pro의 문제 제기 📄
*2026-07-08 · OpenAI Blog*

OpenAI의 새 분석이 인기 코딩 벤치마크 **SWE-Bench Pro의 신뢰성·정확성 문제**를 드러냄 — AI 모델 평가에 대한 우려 제기.

> 🧠 **W27 평가 흐름의 반전**: W27이 "평가를 인프라화하라"(Every Eval Ever)·"도메인 특화 평가"(GeneBench-Pro)였다면, W28은 **"그 벤치마크 자체를 믿을 수 있나"**를 묻는다. [[CogInsight-Generator]] 모델 교체(gpt-4o→GPT-5.6 등)를 리더보드 숫자만 보고 결정하지 말라는 경고 — **우리 도메인(챗봇 시나리오 생성) 전용 회귀 테스트셋**이 결국 답이라는 W27 결론을 강화.

## 🤖 에이전트 (arXiv 멀티에이전트 대량 등장, 07-10)

> 📄 arXiv Multi-Agent Systems 카테고리에 **하루 8편**이 몰림 — "멀티에이전트를 어떻게 **실무에 안전하게 배치**하는가"가 공통 주제. 아래는 [[내-프로필]] 관점 정렬순.

### Who Broke the System? — 멀티에이전트 실패 국소화 📄
*2026-07-10 · arXiv:2607.07989*

LLM 기반 멀티에이전트는 조율된 추론·행동으로 복잡한 문제를 풀지만, **분산 구조 탓에 시스템 수준 실패를 진단하기 어렵다**. 실행이 실패했을 때 "누가(어느 에이전트가) 깨뜨렸나"를 찾는 실패 국소화(failure localization) 연구.

> 🧠 **[[올림푸스-Olympus]] 직결**: 4역할(메티스·헤파이스토스·모모스·헤르메스) Claude 세션이 협업하는 구조에서 "verify.sh가 빨간불인데 어느 세션 탓인지"는 실제 운영 난제. 이 논문은 그 **디버깅 방법론의 학술 근거** — 올림푸스에 "실패 국소화" 관점(단계별 로그·책임 추적)을 넣을 아이디어.

### ASMR — 정비 보고서용 에이전틱 스키마 자동 생성 📄
*2026-07-10 · arXiv:2607.08177*

여러 양식 카테고리에 걸친 **과거 선박 정비·운영 보고서 모음**에서, 핵심을 담는 **간결하고 정보량 높은 스키마를 자동 발견**하는 문제.

> 🧠 **[[CogInsight-Generator]] 시나리오 스키마 생성과 같은 뼈대**: "누적된 실제 문서 → 재사용 가능한 구조(스키마)"는 CogInsight의 **레퍼런스 라이브러리(few-shot·구조 시드)**·Dialog JSON 스키마와 정확히 같은 과제. "과거 시나리오들에서 좋은 스키마를 자동 추출"하는 v0.5+ 아이디어 후보.

### Multi-Agent Firewall — 민감정보 프라이버시 방화벽 📄
*2026-07-10 · arXiv:2607.08282*

LLM을 워크플로우에 안전장치 없이 통합하면 큰 리스크 → **사용자 앞단에서 동작하는 오픈소스 프라이버시 방화벽**을 제안(민감 데이터가 모델로 새는 것을 차단).

> 🧠 **회사 데이터 거버넌스의 정면 해법**: [[GLM-Zhipu-Z-ai]]에서 "중국 모델이라 회사 데이터엔 거버넌스 확인 전까지 부적합"이라 적어둔 바로 그 문제를, **모델 앞단 방화벽**으로 완화하는 접근. 사내 챗봇/자동화에 외부 LLM을 붙일 때의 안전장치 설계 참고.

### Deployment-Time Memorization — 배치 시점의 에이전트 기억 📄
*2026-07-10 · arXiv:2606.10062v2*

파운데이션 모델 에이전트는 **사용자를 넘나들며 기억하는 장수(long-lived) 시스템**이 되고 있어, 기억(memorization)이 모델 가중치의 속성이 아니라 **배치 시점의 명시적 기능**이 된다.

> 🧠 **W27 Memora의 연장선**: "저장·검색을 분리한 에이전트 메모리"(W27) → "배치 시점에 기억을 어떻게 다룰까"(W28). 내가 쓰는 [[Claude-Code-업데이트-동향|Claude Code]] 메모리 기능과 **이 위키 vault의 "한 번 정리해 축적" 철학**이 여전히 학계 최전선과 같은 궤도.

## 🧩 [보강] 에이전트 운영·신뢰성 — 07-08~07-10 수집분

> 🧠 이 절이 이번 보강의 핵심이다. 07-08~07-10 arXiv에는 **"멀티에이전트를 오래·싸게·믿을 수 있게 굴리는 법"** 논문이 몰려 있고, [[올림푸스-Olympus]](4역할 Claude 세션 협업 파이프라인)가 실제로 겪는 문제와 1:1로 대응된다.

### LLM-as-a-Verifier — 검증을 "스케일링 축"으로 승격 📄
*2026-07-08 · arXiv:2607.05391*

사전학습·사후학습·테스트타임 컴퓨트에 이어, **"해답이 옳은지 판정하는 능력(verification)"을 새로운 스케일링 축**으로 규정한 범용 검증 프레임워크.

> 🧠 **[[올림푸스-Olympus]] 모모스(비평가) 역할의 학술 근거**: 올림푸스는 생성자(헤파이스토스)와 **다른 모델**을 비평가로 쓰라고 규약에 박아뒀는데(`config.sh`의 `BUILDER_MODEL`≠비평가 모델), 이 논문은 그 설계가 임기응변이 아니라 **독립적인 스케일링 축**임을 주장한다. "검증에 컴퓨트를 더 쓰면 더 좋아진다"는 명제는 올림푸스 라운드 수·비평 깊이를 조절할 근거.

### Progressive Crystallization — 탐색을 결정론 워크플로우로 "결정화" 📄
*2026-07-09 · arXiv:2607.07052*

IT 운영 에이전트는 **이미 푼 문제도 매번 전체 LLM 추론을 다시 돌리기 때문에 영구적 비용 센터**가 된다. 에이전트의 탐색 결과를 **결정론적·저비용 워크플로우로 굳히는(crystallize) 라이프사이클**을 제안.

> 🧠 **내 자동화 전반의 비용 설계 원칙**: [[올림푸스-Olympus]]가 같은 종류의 작업을 반복 실행할 때, 성공한 경로를 **스크립트/템플릿으로 굳혀** LLM 호출을 빼는 방향. [[mailer|CS SmartHub]]·[[delivery-smarthub]]처럼 "매번 같은 일을 하는" 자동화에도 그대로 적용된다 — LLM은 새 패턴을 만났을 때만 쓰고, 굳은 건 코드로.

### A Workflow-Aware Serving Layer — 노드별 모델 선택 + 검증자 📄
*2026-07-07 · arXiv:2607.02942*

에이전틱 요청은 하나의 워크플로우 = **LLM·도구 호출의 DAG**를 만들고, 각 노드마다 **모델 선택지와 선택적 품질 연산자(verifier)**가 노출된다. 기존 서빙 계층이 다루지 못하는 워크로드로 규정.

> 🧠 **올림푸스 "역할별 모델 분리"의 이론화**: `PLANNER_MODEL=opus / BUILDER_MODEL=sonnet` 식으로 노드마다 다른 모델을 쓰는 내 구성이 정확히 이 논문이 말하는 per-node model choice다. 비용·품질을 **노드 단위로** 튜닝한다는 관점을 명시적으로 가져올 것.

### TeamTR — ⚠ 멀티에이전트가 단일 모델보다 못한 구조적 실패 모드 📄
*2026-07-09 · arXiv:2605.15207*

**"멀티에이전트 LLM이 단일 모델 베이스라인보다 자주 못한다"**는 최근 평가 결과를 인정하고, **공유 컨텍스트 팀을 순차 파인튜닝할 때 생기는 구조적 실패 모드**를 지목 + trust-region 파인튜닝으로 교정.

> ⚠ 🧠 **이번 주 유일한 역방향 신호**: W28 본문 전체가 "멀티에이전트로 가자"인데 이 논문만 **"그래서 정말 더 나은가"**를 묻는다. [[올림푸스-Olympus]]처럼 역할을 쪼갠 구조가 **단일 세션보다 실제로 나은지**를 스스로 측정해본 적 없다는 점에서, 내가 붙잡아둬야 할 반증 후보. (W28 SWE-Bench Pro "벤치마크를 믿지 마라"와 같은 회의주의 계열)

### 상태·메모리 — 긴 작업을 버티는 법 📄
- **StateFuse (2607.05844)** — 에이전트는 분기·재시도·복제를 거치며 **서로 모순되는 관측**을 쌓는데, 대부분의 메모리 계층은 덮어쓰기 규칙 뒤로 그 불일치를 숨긴다. **충돌을 보존하고 검사 가능하게** 만드는 결정론적 메모리. 🧠 올림푸스 `state/` (handoff·progress·critic-feedback)가 라운드마다 덮어써지는 구조와 정면으로 관련 — "이전 라운드 판단과 지금 판단이 왜 다른가"를 남기는 설계.
- **PatchOptic (2607.05483)** — 공유 구조적 상태 위의 워크플로우에서, 컨텍스트 한계 때문에 매 호출에 **필요한 조각만 보여주는(progressive disclosure)** 패턴을 **투영 뷰 + 검증된 구조적 업데이트**로 정식화. 🧠 올림푸스가 역할 세션에 `task.md`·`progress.md`만 잘라 넣는 방식 그 자체.
- **A hierarchical memory architecture (2607.07666)** — LLM의 무상태(stateless) 구조가 **다중 세션 연속성**이 필요한 장기 연구 워크플로우를 가로막는다 → 계층적 메모리로 컨텍스트 한계 돌파. 🧠 이 위키 vault의 "한 번 정리해 축적" 철학과 같은 문제의식.
- **AOI (2512.13956)** — 동적 스케줄링 + **계층적 메모리 압축**으로 클라우드 운영(마이크로서비스 장애 대응) 자동화.
- **Danus (2607.06447)** — **Fact-Graph 메모리**로 수학 추론 에이전트들을 오케스트레이션.

## 의외의 연결점

> 🧠 **W27 → W28 "에이전트 실무 배치" 진행**: W27이 스킬·메모리를 **연구 개념**(SkillOpt·Memora)으로 승격시켰다면, W28은 그것을 **배치 현실**로 끌어내린다 — Deployment-Time Memorization(기억을 배치 기능으로), Multi-Agent Firewall(안전장치), Who Broke the System(실패 진단). "만드는 법"에서 "안전하게 굴리는 법"으로 초점 이동. [[올림푸스-Olympus]] 운영(실패 국소화·verify 게이트)이 바로 이 지점.
> 🧠 **평가 신뢰성의 역설**: W27 "평가를 인프라화·도메인 특화하라" → W28 "그런데 그 벤치마크(SWE-Bench Pro) 믿을 수 있나". 결론은 동일 — **내 도메인 전용 회귀 테스트셋**([[CogInsight-Generator]] 시나리오 생성 품질)이 리더보드보다 낫다.
> 🧠 **"스키마/구조 자동 생성" 3중 교차**: ASMR(과거 보고서→스키마) + Flint(compact 스펙→표현력 있는 시각화) + CogInsight v0.4.0(레퍼런스→다이어그램). "누적 자료에서 재사용 구조를 뽑아 시각화한다"가 내 프로젝트의 다음 축과 학계 흐름에서 동시에 떠오름.
> 🧠 **음성 채널 2주 연속**: W27 Gemma 4 실시간 음성(오픈) → W28 GPT-Live(상용). 텍스트 챗봇의 다음 채널로서 음성이 양 진영에서 굳어지는 중.
> 🧠 **비개발→개발의 시대적 정당화**: MIT "비기술 인력의 챗봇 SW 제작" + ChatGPT Work(비개발자도 목표만 주면 완성) → [[내-프로필]]의 전환 서사가 개인 노력이 아니라 **업계가 밀어주는 방향**임을 확인.
> 🧠 **(보강) "에이전트를 굳혀라"가 이번 주 최대 실무 메시지**: Progressive Crystallization(탐색→결정론 워크플로우) + From Atomic Actions to SOPs(원자 도구→표준절차) + MUSE-Autoskill(스킬을 평가 루프에) 세 편이 **같은 말을 다른 층위에서** 한다 — **"LLM이 한 번 푼 것을 다시 LLM으로 풀지 마라."** 내 자동화([[mailer|CS SmartHub]]·[[올림푸스-Olympus]])의 비용·안정성 원칙으로 그대로 채택할 만한 결론.
> ⚠ 🧠 **(보강) 멀티에이전트 회의주의 2종**: TeamTR("멀티에이전트가 단일 모델보다 못한 경우가 있다") + SWE-Bench Pro("벤치마크를 믿지 마라")가 같은 주에 나왔다. 역할을 쪼갠 [[올림푸스-Olympus]] 구조가 **단일 세션 대비 실제로 나은지** 내가 측정한 적 없다는 사실을 남겨둔다 — 이번 주 자료가 던진 가장 불편한 질문.
> 🧠 **(보강) 프라이버시 "안팎" 대칭**: W28 본문의 Multi-Agent Firewall(민감정보가 **외부 모델로** 새는 것)과 보강분의 PiSAs(정보가 **조직 내부 다른 사용자에게** 새는 것)가 한 쌍. 사내 공용 챗봇은 두 방향을 **동시에** 막아야 한다는 뜻 — [[GLM-Zhipu-Z-ai]] 거버넌스 메모에 더할 관점.
> 🧠 **(보강) "조직 기억"이 내 위키의 학술적 이름**: Organizational Memory(2607.03228)·계층적 메모리(2607.07666)·StateFuse(충돌 보존)가 공통으로 말하는 건 **"쌓인 맥락을 검사 가능한 구조로 유지하라"** — 이 vault([[index]])가 개인 스케일에서 하고 있는 일과 정확히 같다.
