## 🔥 빅 이슈

### ChatGPT Work — "몇 시간이고 프로젝트를 붙잡는 행동 에이전트" 📄
*2026-07-09 · OpenAI Blog*

**ChatGPT Work**는 **앱과 파일을 가로질러 행동(action)하고, 필요하면 몇 시간이고 한 프로젝트에 머물며, 목표를 완성된 결과물로 바꾸는 에이전트**. "가장 야심 찬 일의 파트너"로 포지셔닝.

> 🧠 **W26 CUGA·W27 MIT 에이전트 Q&A의 제품화 지점**: "에이전트가 실무를 대신한다"가 연구·데모를 넘어 **오피스 생산성 제품**으로. 내 [[에이전트-자동화-도구]] 정리(workflows·/goal·routines)와 같은 계열이고, [[mailer|CS SmartHub]]처럼 "여러 도구를 가로질러 하나의 결과물을 만드는" 내 자동화 방향과 정확히 겹친다. CS팀 반복 업무를 "몇 시간짜리 완성 작업"으로 위임하는 시나리오를 상상해볼 지점.

### GPT-5.6 — 프런티어 모델 + M365 Copilot 기본 채택 📄
*2026-07-09 · OpenAI Blog*

**GPT-5.6**: "토큰당 더 많은 지능, 달러당 더 강한 성능, 필요 시 더 큰 능력". 같은 날 **Microsoft 365 Copilot의 선호(preferred) 모델**로 채택 — Word·Excel·PowerPoint·Chat·Cowork 전반을 구동.

> 🧠 **[[CogInsight-Generator]] 모델 업그레이드 후보**: 현재 gpt-4o로 6단계 생성 파이프라인을 돌리는데, "성능/달러" 개선이 셀링포인트인 GPT-5.6은 **비용·품질 재검토 대상**(W27 "최신·최대 ≠ 최적" 기조와 함께 저울질). M365 Copilot 기본 채택은 내가 쓰는 [[Teams-Gmail-캘린더-Gemini-연동|M365·Cowork]] 환경에도 직접 영향.

### 코딩 벤치마크의 신호와 잡음 — SWE-Bench Pro의 문제 제기 📄
*2026-07-08 · OpenAI Blog*

OpenAI의 새 분석이 인기 코딩 벤치마크 **SWE-Bench Pro의 신뢰성·정확성 문제**를 드러냄 — AI 모델 평가에 대한 우려 제기.

> 🧠 **W27 평가 흐름의 반전**: W27이 "평가를 인프라화하라"(Every Eval Ever)·"도메인 특화 평가"(GeneBench-Pro)였다면, W28은 **"그 벤치마크 자체를 믿을 수 있나"**를 묻는다. [[CogInsight-Generator]] 모델 교체(gpt-4o→GPT-5.6 등)를 리더보드 숫자만 보고 결정하지 말라는 경고 — **우리 도메인(챗봇 시나리오 생성) 전용 회귀 테스트셋**이 결국 답이라는 W27 결론을 강화.

### GPT-Live — 자연스러운 인간-AI 상호작용을 위한 음성 모델 📄
*2026-07-08 · OpenAI Blog*

**GPT-Live**: ChatGPT Voice를 구동하는 새 세대 음성 모델.

> 🧠 **W27 Gemma 4 실시간 음성(Cerebras)의 상용 대응판**: 오픈 모델 진영(Gemma+Cerebras)에 이어 OpenAI도 음성 채널 강화. 텍스트 챗봇의 다음 채널이 음성이라는 W27 판단이 2주 연속 확인됨 — [[CogInsight-Generator]]류 시나리오 도구가 언젠가 "음성 시나리오"로 확장될 근거.

## 🤖 에이전트 (arXiv 멀티에이전트 대량 등장, 07-10)

> 📄 arXiv Multi-Agent Systems 카테고리에 **하루 8편**이 몰림 — "멀티에이전트를 어떻게 **실무에 안전하게 배치**하는가"가 공통 주제. 아래는 [[내-프로필]] 관점 정렬순.

### Who Broke the System? — 멀티에이전트 실패 국소화 📄
*2026-07-10 · arXiv:2607.07989*

LLM 기반 멀티에이전트는 조율된 추론·행동으로 복잡한 문제를 풀지만, **분산 구조 탓에 시스템 수준 실패를 진단하기 어렵다**. 실행이 실패했을 때 "누가(어느 에이전트가) 깨뜨렸나"를 찾는 실패 국소화(failure localization) 연구.

> 🧠 **[[올림푸스-Olympus]] 직결**: 4역할(메티스·헤파이스토스·모모스·헤르메스) Claude 세션이 협업하는 구조에서 "verify.sh가 빨간불인데 어느 세션 탓인지"는 실제 운영 난제. 이 논문은 그 **디버깅 방법론의 학술 근거** — 올림푸스에 "실패 국소화" 관점(단계별 로그·책임 추적)을 넣을 아이디어.

### ASMR — 정비 보고서용 에이전틱 스키마 자동 생성 📄
*2026-07-10 · arXiv:2607.08177*

여러 양식 카테고리에 걸친 **과거 선박 정비·운영 보고서 모음**에서, 핵심을 담는 **간결하고 정보량 높은 스키마를 자동 발견**하는 문제.

> 🧠 **[[CogInsight-Generator]] 시나리오 스키마 생성과 같은 뼈대**: "누적된 실제 문서 → 재사용 가능한 구조(스키마)"는 CogInsight의 **레퍼런스 라이브러리(few-shot·구조 시드)**·Dialog JSON 스키마와 정확히 같은 과제. "과거 시나리오들에서 좋은 스키마를 자동 추출"하는 v0.5+ 아이디어 후보.

### Multi-Agent Firewall — 민감정보 프라이버시 방화벽 📄
*2026-07-10 · arXiv:2607.08282*

LLM을 워크플로우에 안전장치 없이 통합하면 큰 리스크 → **사용자 앞단에서 동작하는 오픈소스 프라이버시 방화벽**을 제안(민감 데이터가 모델로 새는 것을 차단).

> 🧠 **회사 데이터 거버넌스의 정면 해법**: [[GLM-Zhipu-Z-ai]]에서 "중국 모델이라 회사 데이터엔 거버넌스 확인 전까지 부적합"이라 적어둔 바로 그 문제를, **모델 앞단 방화벽**으로 완화하는 접근. 사내 챗봇/자동화에 외부 LLM을 붙일 때의 안전장치 설계 참고.

### Deployment-Time Memorization — 배치 시점의 에이전트 기억 📄
*2026-07-10 · arXiv:2606.10062v2*

파운데이션 모델 에이전트는 **사용자를 넘나들며 기억하는 장수(long-lived) 시스템**이 되고 있어, 기억(memorization)이 모델 가중치의 속성이 아니라 **배치 시점의 명시적 기능**이 된다.

> 🧠 **W27 Memora의 연장선**: "저장·검색을 분리한 에이전트 메모리"(W27) → "배치 시점에 기억을 어떻게 다룰까"(W28). 내가 쓰는 [[Claude-Code-업데이트-동향|Claude Code]] 메모리 기능과 **이 위키 vault의 "한 번 정리해 축적" 철학**이 여전히 학계 최전선과 같은 궤도.

### 그 외 멀티에이전트 단신 📄
- **Collective Intelligence with Foundation Models (2607.07729)** — 여러 모델을 협력 추론 시스템으로 조율(solver가 독립 초안 생성) → 더 안전·신뢰. 🧠 "제안-검증 구조"(W25)·올림푸스 다역할 협업과 같은 철학.

- **CPM 기반 감정 평가 멀티에이전트 (2607.07824)** — 페르소나 대화 에이전트의 **동적 감정 변화**(상담·CS·교육·스토리텔링). 🧠 챗봇 시나리오의 **감정/톤 설계** 관점.
- **Communication Gating in Cooperative VLN (2607.08504)** — 대역폭 제한 협력 시각-언어 내비게이션에서 통신을 언제 열고 닫나. (연구 단신)
