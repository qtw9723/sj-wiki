> [!tip] 핵심 takeaway
> **너의 핵심 정체성("주어진 업무를 자동화하며 개발자로 성장")을 가장 직접적으로 강화해주는 도구군.**
> 과거: 파이썬+크론탭+메일([[내-프로필]]) → 현재: [[schedule-reporter-kakao]] → 다음: 아래 Claude Code 자동화 기능들. **같은 본능을 점점 더 강력한 도구로 푸는 중**이다.
> 당장 쓸 것 3가지: ① **dynamic workflows**(대규모 일괄 작업) ② **`/goal`**(조건 충족까지 자동 반복) ③ **routines/`/schedule`·`/loop`**(주기 실행). 기능 출처·주차는 [[Claude-Code-업데이트-동향]].

## 자동화 스펙트럼 (작은 것 → 큰 것)
| 도구 | 무엇 | 네 활용 시나리오 |
|---|---|---|
| `/loop`, Monitor (W15) | 프롬프트 주기 실행 / 이벤트 감시 | **(지금 이 위키)** raw→wiki 정리 자동화 |
| routines, `/schedule` (W16) | 클라우드 크론 에이전트 | [[schedule-reporter-kakao]]식 정기 리포팅 |
| `/goal` (W20) | 완료 조건까지 자동 반복 | [[Cogi-POC-Generator]] "테스트 통과까지" 개발 |
| `claude agents` (W20) | 여러 세션 한 화면 관리 | 여러 작업 병렬 진행 |
| dynamic workflows (W22) | 수십~수백 서브에이전트 오케스트레이션 | [[공통-기술스택]] 전 프로젝트 일괄 마이그레이션 |
| auto mode (W13~21) | 권한 프롬프트를 안전 검사로 대체 | 반복 승인 제거 |

## "자동화의 마지막 단계 = 알림"
- 네가 [[schedule-reporter-kakao]]에서 카카오로 결과를 밀어주듯, Claude Code도 **mobile push notifications(W16)**, **terminalSequence 알림 훅(W20)**을 제공.
- 공통 원리: *수집 → 처리 → 사람에게 전달*. 자동화 설계 시 "어디로 알릴지"를 항상 마지막에 설계할 것.

## 에이전트 설계로 확장
- [[Cogi-POC-Generator]]가 "자연어→Dialog JSON" LLM 도구로 커지면, 이는 곧 **에이전트/툴 정의** 문제가 된다 → [[claude-api]](tool use, 구조화 출력, MCP) 참고가 필요해지는 지점.
- zod([[공통-기술스택]])로 LLM 출력 검증 = 에이전트 신뢰성 확보의 정석.

## 의외의 연결점
- **SM 시절의 크론·네트워크 운영 감각([[내-프로필]])**이 여기서 빛난다: 스케줄링·모니터링·알림은 SM 업무의 본질이었고, 지금 도구만 AI로 바뀐 것. 너는 자동화 전환에 이미 직관이 있다.
- **회사 업무에도 같은 패턴**: 본업의 "정해진 시각 자동 발송·상태 판정" 류 작업도 전형적 크론/모니터 → routines·`/schedule`·Monitor와 동형. 자동화 설계 직관이 그대로 전이.
- **MCP 커넥터 환경**: 네가 실제 쓰는 scheduled-tasks·computer-use 등은 [[내-MCP-커넥터-환경]] 참고.
