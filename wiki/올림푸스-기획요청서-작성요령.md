> [!tip] 핵심 takeaway
> 🧠 [[올림푸스-Olympus]]에 넣는 기획서는 **"사람에게 설명하는 일반 계획서"가 아니라 "기획자(메티스)가 분석·분해·개발·검증·배포까지 자동으로 굴리도록 지시하는 요청서"**다. 그래서 ① **수용 기준은 테스트로 판정 가능**하게 ② **가정 불가한 결정은 "진짜 블로커"로 명시** ③ **검증·배포 방법을 함께** 적는 게 핵심.
> 🧠 잘 쓴 요청서 = 사람 개입 최소화. 애매하면 메티스가 "합리적 가정"으로 그냥 진행해버리므로, **꼭 사람이 정해야 하는 것만 블로커로** 콕 집어야 폰으로 물어본다.

## 이 문서의 용도
📄 `spec/<프로젝트>/*.md`에 넣을 기획요청서를 이 요령대로 작성한다. 넣고 `bash run.sh` 하면 [[올림푸스-Olympus]]가 분석→(블로커 폰승인)→구현→verify→(최종 폰승인)→자동머지→배포→아카이브까지 굴린다.

---

## 1. 메티스가 요청서를 어떻게 읽는가 (왜 이렇게 써야 하나)
🧠 이해하고 쓰면 실수가 준다.

- **요구사항 분석**: 요청서를 읽고 `requirements.md`로 분석 → 작은 작업들(`backlog.md`)로 분해한다. → **목적·요구·범위가 명확할수록 분해가 정확**.
- **모호함 처리**: 애매한 건 **멈추지 않고 "합리적 가정"으로 진행**하고 `open-questions.md`에 쌓아 끝에 한 번에 보고한다. → **가정되면 곤란한 건 반드시 "블로커"로 못 박아야** 폰으로 물어본다.
- **그라운드 트루스**: LLM 합의가 아니라 **실행되는 게이트(`VERIFY_CMD`: lint·test·build)**가 심판. → **수용 기준을 테스트로 판정 가능**하게 써야 한다.
- **기존 코드 보존(brownfield)**: 기존 repo면 최소 변경·동작 보존 원칙. **기존 테스트 green 유지**가 기본 수용 기준.

## 2. 필수 구성요소 (체크리스트)
> [!info] 이 7개가 있으면 사람 개입 없이 잘 굴러간다
> - [ ] **목적/배경** — 무엇을·왜 (한두 문단)
> - [ ] **대상 repo** — 어느 프로젝트를 고칠지 (폴더명 = repo명, 아래 §4 주의)
> - [ ] **요구사항 명세** — 구체적 동작 + **입력→출력 예시**
> - [ ] **수용 기준(AC)** — **테스트로 판정 가능**하게. 기존 테스트 green 유지 포함
> - [ ] **⚠️ 진짜 블로커** — 가정 불가·반드시 사람이 정해야 하는 값 (없으면 "없음"이라고)
> - [ ] **범위 밖(YAGNI)** — 하지 말 것
> - [ ] **검증·배포** — `VERIFY_CMD`와 배포 규약(명령)

## 3. 복붙 템플릿
```markdown
# <프로젝트/기능 제목>

## 목적
<무엇을, 왜. 한두 문단. 사용자·시나리오.>

## 대상
- repo: <예: notepad → ~/IdeaProjects/notepad>
- 수정 파일(예상): <예: src/lib/tags.js, 그 테스트>
- 이 범위 밖 파일은 건드리지 않는다.

## 요구사항 (동작 명세)
1. <구체적 규칙 1> — 예: `'#work'` → `'work'`
2. <구체적 규칙 2> — 예: `'Side Project'` → `'side-project'`
3. 처리 순서/엣지케이스: <...>

## ⚠️ 반드시 사람에게 확정받아야 하는 값 (가정 금지 — 진짜 블로커)
- <이게 없으면 개발 시작 불가하고, 임의 가정하면 안 되는 것.
  예: 기존 저장 데이터와 호환돼야 하는 포맷·구분자, 외부 연동 자격증명, 핵심 데이터모델 선택>
- (해당 없으면: "없음 — 모두 합리적 가정으로 진행 가능")

## 수용 기준 (그라운드 트루스)
- `<VERIFY_CMD>` 전부 통과 (예: npm run verify = lint + test + build)
- <신규 동작>을 검증하는 테스트가 추가되어 통과
- 기존 테스트 전부 green 유지(회귀 없음)

## 범위 밖 (하지 말 것)
- <YAGNI 항목들>

## 배포
- <배포 명령 한 줄. 예: git push origin main → Vercel 자동배포>
- (배포 안 하거나 규약 없으면 생략 → 올림푸스가 완료 시 물어본다)
```

## 4. 작성 팁 & 함정
🧠 이번에 실제로 굴려보며 얻은 것들.

- **대상 repo는 되도록 명시**: spec 폴더가 여러 개면 올림푸스가 **알파벳순 첫 폴더**를 고른다(예: `Mindboard`가 `notepad`보다 앞섬). 실행 시 `TARGET_REPO=...`로 지정하거나 spec 폴더를 하나만 두는 게 안전. 📄
- **"진짜 블로커"는 메티스 기준에 맞춰 표현**: 메티스는 *"개발을 시작조차 못 하고, 가정으로 대체 불가한 것(핵심 데이터모델·외부 자격증명)"* 일 때만 멈춘다. 단순 취향 선택은 가정으로 진행해버림 → 블로커로 만들려면 **"기존 데이터와 호환돼야 함 / 틀리면 깨짐 / 나는 알 수 없음"** 같은 **불가역·비가정성**을 근거로 적어라. 📄
- **AC는 반드시 실행 검증 가능하게**: "잘 되게" 말고 `입력 X → 출력 Y`처럼. 그래야 verify가 심판이 된다.
- **배포 규약은 명령 그대로**: 메티스가 그 명령줄을 `state/deploy.md`로 뽑아 머지 후 실행한다. 안 적으면 완료 시 폰/터미널로 "배포 명령 입력 또는 skip"을 묻는다. 📄
- **범위를 좁게**: 한 요청서 = 작은 목표 하나. 크면 여러 spec으로 나눠라(각각 분석·개발·아카이브 사이클).

## 5. 완료되면 자동으로 (요청서에 안 써도 됨)
📄 올림푸스가 알아서 하는 것들 — 요청서는 "무엇을"만, "어떻게 마무리"는 파이프라인 담당:
- 각 작업 green → 작업 브랜치에 자동 커밋
- 전체 완료 → 최종 verify → **폰 승인**(olympus-inbox + Gmail 버튼 메일) → 자동 머지 → 배포 → 기획서를 `spec/_done/`으로 아카이브 + state 초기화
- 도중 블로커/의도질문도 폰으로 옴 → GitHub 앱에서 답하면 이어감 ([[올림푸스-Olympus]] 원격 결정 브리지)

## 6. 실행 (참고)
```bash
cd ~/IdeaProjects/olympus
bash reset.sh                              # 새 요청서면 state 초기화
TARGET_REPO=~/IdeaProjects/<repo> \
REMOTE=1 POLL_TIMEOUT_SEC=1800 \
VERIFY_CMD="<검증 명령>" \
  bash run.sh
```
- `REMOTE=1` = 개입을 폰으로. 끄면 터미널에서 처리.
- 배포/머지/아카이브 자동. 끄려면 `AUTO_MERGE=0` / `DEPLOY_ON_DONE=0` / `ARCHIVE_ON_DONE=0`.
- **토큰 소진 자동 대기·재개**(기본 켜짐, `TOKEN_WAIT=1`) 📄: 장시간 무인 실행 중 Claude Code **사용량 한도**에 걸려도 자동으로 리셋까지 기다렸다 그 자리에서 이어간다 → **요청서에 쓸 필요 없는 런타임 기능**(그냥 두면 됨). 끄려면 `TOKEN_WAIT=0`, 재확인 간격은 `TOKEN_RETRY_SEC`(기본 1200초). 대기 중 토큰 ≈0. 단 프로세스가 죽으면 유실 → 다시 `bash run.sh`(state에서 재개). 상세: [[올림푸스-Olympus]] §헤드리스 무인 실행.
