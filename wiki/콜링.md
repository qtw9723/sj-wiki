## 한 줄 정의
📄 **콜링(Calling)** — 교회에서 쓸 **일정 공유 캘린더** — 여러 관리자가 일정을 등록하고, 사용자는 자신이 속한 **카테고리(팀/그룹)** 의 일정만 골라 본다. 로그인·권한 관리 필수, **1차는 대학부** 대상, 추후 **모바일 앱·ERP** 확장 대비.

## 문제 / 배경
- 📄 교회 일정이 단체 메신저·주보·구두 등 분산 채널로 공유돼 누락·중복·장소 충돌 발생. (요구사항 기반 정리)
- 🧠 소속에 따라 봐야 할 일정이 다른데 모두 섞여 전달 → 정작 본인 관련 일정을 찾기 어려움 (배경 추론).

## 핵심 요구사항 (사용자 명시) 📄
1. 핵심 기능은 **일정 공유용 캘린더**.
2. **여러 관리자**가 직접 일정을 등록, **여러 사용자**가 조회.
3. 사용자·관리자는 **카테고리(팀/그룹)별**로 일정을 등록하고, 해당 카테고리 소속자에게만 그 일정이 보인다.
4. **모든 사용자 로그인 필수**, **권한 관리 화면** 필요.
5. 추후 **모바일 앱**(일정 등록·변경 시 알림) 및 **ERP(인력·재정/회계 제외)** 확장 → **확장성 있게** 설계.
6. 스택: **웹 표준(React + Node/Spring)**(협의 후 변경 가능), 규모: **소규모(~수백 명)**.
7. **1차 적용 대상은 대학부**, 검증 후 가용성에 따라 교회 전체로 확장.
8. 추후 **영혼관리시스템 연동**을 위해 개인 식별 기본정보를 수집·매핑.

## 권한 · 카테고리 설계 (핵심)
- 📄(확정) **권한은 카테고리별로 부여**된다. 같은 사용자가 A 카테고리에서는 일반 사용자, B 카테고리에서는 관리자일 수 있다.
- 📄(확정) **권한 부여 주체**: 전체 관리자 = 모든 카테고리에 모든 권한 부여 가능 / 카테고리 관리자 = 자신이 관리하는 카테고리의 일반 사용자 등록·관리.
- 🧠 설계안: ① **카테고리 스코프 역할**(기본 축, category_membership.role_in_category=MEMBER/ADMIN) + ② **글로벌 역할**(ADMIN/SUPER_ADMIN, 시스템 전반).
- 🧠 **가시성 규칙**: 일반 사용자 = 본인 소속 카테고리 + 전체 공개 / 카테고리 관리자 = 담당 카테고리(편집) + 본인 소속 / 전체 관리자 = 전부. **서버측 강제 필터링**(프론트 신뢰 X).
- 🧠 **데이터 모델 핵심**: user(+식별 기본정보: name·birth_date·phone·soul_mgmt_ref) · role · permission · category(self-ref) · category_membership(다대다) · event · event_category(다대다) · event_occurrence. 시각 UTC 저장→KST 표시.

## 핵심 강점 — 관리 용이성 (전체 일정 조율) ★ 📄(사용자 강조 2026-07-01)

📄(강조 확정) 이 앱의 **가장 큰 강점 중 하나는 관리자의 일정 조율이 쉬워진다는 점**이다. 관리자가 자신이 챙겨야 할 여러 팀 일정을 **팀별로 한눈에** 확인하고, 전체 일정을 겹침 없이 조율할 수 있게 하는 것이 핵심 가치다.

**지금(문제)** 🧠: 관리자는 여러 카톡방·주보·구두 공지를 **머릿속에서 취합**해야 한다 → 놓치고, 겹치고, 장소가 충돌한다. 조율 부담이 사람에게 있다.

**이 앱(해결)** — 조율 부담을 **시스템이 대신** 진다:
- 📄 **팀별 필터로 골라 보기**: 사이드바에서 담당·관심 있는 카테고리만 켜서 본다. 여러 채널을 오갈 필요 없이 캘린더 하나로 취합.
- 📄 **색상 구분**: 일정 칩이 팀 색상이라 "어느 팀 일정인지" 즉시 식별. 셀에 몰리면 "+더보기"로 정돈.
- 🧠 **충돌 즉시 발견**: 전체 관리자·교역자는 모든 팀 일정을 한 화면에 겹쳐 보므로, 같은 시간·같은 장소의 중복 모임을 **등록 시점에 바로** 알아챈다(사후 수습 X).
- 📄 **관리자만 등록 = 단일 출처(single source of truth)**: 일정이 흩어지지 않고 캘린더 한 곳에 모여 누락·중복이 구조적으로 준다.
- 📄 **마이페이지 역할 배지**: "내가 관리하는 팀"이 명확히 보여 관리 대상이 흩어지지 않는다.
- 🧠 **확장 시 강화**: 로드맵의 [[#확장 로드맵|자원·장소 예약(Phase3)]]까지 가면 장소 중복 예약 감지로 조율이 더 강력해진다.

🧠 **발표용 한 줄**: "여러 카톡방을 뒤지며 사람이 맞추던 일정 조율을, **캘린더 하나에서 팀별로 골라 보며** 겹침 없이 관리한다."

## 영혼관리시스템 연동 (향후) 📄(확정)

- 당장은 캘린더 내부에서만 사용하고, 추후 **영혼관리시스템이 준비되면 식별정보 기준으로 매핑**하여 연동(soul_mgmt_ref에 외부 키).
- 🧠 개인정보 최소 수집·접근통제·동의 필요.

  3. **성도 소통&기도 나눔** — 공지 일괄 전달, **기도제목 나눔**, 자동 리마인더. → 📄(처리 확정 2026-07-01) 이 소통·기도 기능은 **1차 MVP가 아니라 [[#확장 로드맵|Phase2.5 로드맵]]으로 이동**. 기능①의 **절기 표시는 1차 포함**(관리자 반복 일정 설정 방식). ↔ 위 「기능 범위」 표·처리 콜아웃 참조.
- 📄 **03 기대 효과**: 사역적 효과(참여↑·누락↓) / 운영적 효과(반복 안내↓·일정 정확도↑). 제안서에 "위 화살표·효과는 예시·가정이며 실제 수치는 운영 데이터로 확인"이라는 단서 명시.

- 📄 **05 마무리·요청사항**: 요청①"콜링 개발·도입 추진에 대한 리더십의 승인" 요청②"대학부 시범 운영(파일럿) 허가".
- 🧠 **"기존 서버" 표현 관련**: 제안서는 "자체 제작·기존 서버 활용"이라 적었는데, 이건 [[#비용 · 배포 (운영) 📄(확정 2026-07-01, 사용자)|비용·배포 섹션]]의 "1차=개인 서버 무료"와 같은 뜻으로 보인다(발표자 개인이 이미 가진 서버 = "기존 서버"). 다만 제안서만 보면 "교회의 기존 서버"로 오해될 소지도 있어 ⚠️ 발표 시 "제 개인 서버"라고 구두로 명확히 하는 게 안전.

## 제안 🧠
- 1차는 범위를 **로그인·카테고리·권한·일정 CRUD·반응형 웹**으로 못 박고, 알림/자원예약은 1.5~2차로 미뤄 MVP 속도 확보.
- 📄 **로그인 = 이름 + 소속기관 + 생년월일 + 비밀번호 + 관리자 승인제**(2026-07-01 정정, 소셜 미지원). 🧠 **비밀번호로 인증 강도를 확보**하고 **관리자 승인 게이트**로 미검증 가입을 차단하는 이중 안전장치 — 교인 대상이라 진입 장벽도 낮음. 비밀번호 분실은 관리자 재설정. (v1.1에서 '고령 접근성' 별도 항목은 삭제됨 — 일반 접근성은 화면 공통 요구로 유지.)
