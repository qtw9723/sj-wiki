## 🤖 에이전트·멀티에이전트

### WorkBench 2026: 직장 에이전트 2년 후 재측정 📄
*2026-06-15 · arXiv 2606.13715*

2024년 WorkBench 벤치마크 재측정. 2024년 최고 모델(GPT-4)이 43% 달성, 유해 행동 26%. **2026년 최고 모델: Claude Opus** — 점수 대폭 상승, 단 한계 여전.

> 🧠 에이전트가 2년간 크게 발전했지만 여전히 실수율 존재. 자동화 파이프라인 설계 시 **사람 검토 포인트** 필수.

### MASLab: LLM 멀티에이전트 통합 코드베이스 📄
*2026-06-15 · arXiv 2505.16988*

LLM 기반 멀티에이전트 시스템(MAS) 비교·실험을 위한 **통합 코드베이스** 공개. 다양한 프레임워크를 단일 인터페이스에서 실험 가능.

> 🧠 LangGraph·AutoGen·CrewAI 비교 학습 시 시작점으로 유용. [[에이전트-자동화-도구]]와 연결해 참조할 실습 자료.

### GUI 에이전트의 시각적 메모리 한계 연구 📄

GUI 에이전트가 화면 메모리에 의존할 때 발생하는 실패 패턴 체계적 분석. 경험 기반 메모리 활용 에이전트 도입 시 주의할 실패 케이스 정리.

> 🧠 자동화 스크립트에 GUI 에이전트 도입 전 실패 모드 파악 필수.

### HuggingFace Spaces 에이전트 체이닝 — 3D 파리 갤러리 📄

두 개의 HuggingFace Spaces를 에이전트가 체이닝하여 3D 파리 갤러리를 자동 생성한 사례. 에이전트가 외부 도구를 순차 호출하는 패턴 시연.

> 🧠 [[Cogi-POC-Generator]]에서 복수 LLM·도구 체이닝 구현 시 직접 참조 가능한 패턴.

### 게임 이론적 멀티에이전트 제어 — 프롬프트 인젝션 방어 📄
*2026-06-15 · arXiv 2606.10322*

멀티턴 LLM의 컨텍스트 오염(프롬프트 인젝션·포이즈닝) 취약점에 게임 이론 기반 멀티에이전트 제어로 대응하는 연구.

> 🧠 챗봇 시나리오 구성 시 악의적 입력 방어 패턴. 보안 요구사항 있는 CS 챗봇 배포 시 참조.

### Intelligence Entropy — 프로덕션 멀티에이전트 드리프트 원리 📄
*2026-06-17 · arXiv 2606.18065*

LLM 멀티에이전트 시스템이 프로덕션으로 전환 시 비선형적으로 성능이 저하되는 현상을 **Intelligence Entropy(지능 엔트로피)** 원리로 공식화. ADE 안정성 프레임워크 제안.

> 🧠 클라우드 루틴([[웹-크롤링-기초]])처럼 장기 운영 에이전트는 시간이 지날수록 드리프트한다는 경고. 주기적 검증 루틴의 필요성.

### 멀티에이전트 LLM 동시성 이상 검증 탐지·방지 📄
*2026-06-17 · arXiv 2606.17182*

여러 에이전트가 공유 메모리·벡터 인덱스·도구 레지스트리를 동시 접근할 때 발생하는 **동시성 이상(경쟁 조건)**을 검증 기법으로 탐지·방지하는 연구.

> 🧠 멀티에이전트 시스템 설계 시 상태 공유 구조의 안전성 고려 필수.

### Strands Agents + LeRobot — HuggingFace Hub에서 로봇 하드웨어로 📄
*2026-06-17 · HuggingFace Blog*

Amazon Strands Agents와 HuggingFace LeRobot을 연결해 Hub의 모델을 실제 로봇 하드웨어에 바로 배포하는 파이프라인 시연.

> 🧠 에이전트 체이닝이 소프트웨어를 넘어 물리 세계로. [[에이전트-자동화-도구]] 확장 방향.

### ProvenanceGuard — MCP 기반 LLM 에이전트 출처 검증 📄
*2026-06-17 · arXiv 2606.18037*

MCP(Model Context Protocol)로 다양한 소스(검색·API·DB·의료기록)를 호출하는 LLM 에이전트의 **출처 인식 사실성 검증** 프레임워크.

> 🧠 **직접 연관**: [[내-MCP-커넥터-환경]]에서 NaverSearch·KakaoMap·PlayMCP 등 외부 소스를 MCP로 호출 중. 에이전트가 외부 정보를 인용할 때 출처 검증이 핵심.

### LegalHalluLens — 타입별 환각 감사 + 보정된 멀티에이전트 토론 📄
*2026-06-17 · arXiv 2606.18021*

법률 워크플로우에 투입된 AI의 환각률이 집계 지표상 **~52%**에 달하지만, 평균값이 "오류가 어디에·어느 방향으로 몰리는지"를 가려버린다고 지적. 타입별로 환각을 감사하고, 보정된(calibrated) 멀티에이전트 토론으로 신뢰성을 높이는 접근.

> 🧠 **챗봇 신뢰성 직결**: "평균 정확도"가 아니라 **오류가 집중되는 케이스를 분리**해 봐야 한다는 통찰. [[Cogi-POC-Generator]] 시나리오 QA에서 "어떤 질문 유형에서 틀리는가"를 구분 측정하는 관점. [[#ProvenanceGuard — MCP 기반 LLM 에이전트 출처 검증|ProvenanceGuard]]·Deployment Simulation과 함께 "에이전트 신뢰성 검증" 묶음.

### AI 에이전트 네트워크 신뢰성 — 제안-검증 구조 최적화 📄
*2026-06-17 · arXiv 2606.18121*

단일 모델 호출이 아니라 **여러 불완전한 에이전트의 협업**(일부는 해답 조각을 제안, 일부는 검증, 결과를 종합)이 단일 모델을 능가하는 구조를, 통신 이론의 밀도 진화(density evolution)·정지 집합(stopping set)으로 분석해 아키텍처를 최적화.

> 🧠 **"제안자 + 검증자 + 종합자"가 단일 호출보다 낫다**는 정량적 근거. 멀티에이전트 파이프라인 설계 시 검증 노드를 어디에 둘지 판단 기준. 동시성 이상(2606.17182)·Intelligence Entropy와 같은 "프로덕션 멀티에이전트 신뢰성" 계열.

### Tacit Coordination — 통신 없는 LLM 협응 📄
*2026-06-17 · arXiv 2601.22184 (v2)*

LLM들이 **명시적 통신 없이** 조율해야 하는 멀티에이전트 상황(인간-AI 상호작용, 안전 크리티컬 시나리오)에서 인간처럼 암묵적 협응(tacit coordination)을 할 수 있는지 연구.

> 🧠 여러 에이전트/도구가 직접 메시지를 주고받지 않고도 공통 목표로 수렴하는 설계. 자동화 파이프라인에서 에이전트 간 결합도를 낮추는 패턴으로 참조.

### DRFLOW — 개인화 워크플로우 예측 딥리서치 벤치마크 📄
*2026-06-17 · arXiv 2606.18191*

기존 딥리서치(DR) 시스템이 "보고서·요약 생성"에 치중하는 반면, 많은 기업 업무는 에이전트가 **구체적인 워크플로우(해야 할 일의 절차)를 식별**하길 요구한다는 문제의식에서 출발한 벤치마크.

> 🧠 **업무 자동화 직결**: "요약을 내놓는 AI"가 아니라 "다음에 뭘 어떤 순서로 해야 하는지 짚어주는 AI". [[내-프로필]]의 반복 업무 자동화 지향과 정확히 맞닿음. CS 업무 절차를 에이전트가 예측·제안하는 방향.

### Trustworthy BDaaS — LLM 오케스트레이션 데이터 엔지니어링/MLOps 멀티에이전트 📄
*2026-06-17 · arXiv 2606.17915*

Big-Data-as-a-Service 플랫폼의 전 과정(수집·정제·피처 엔지니어링·모델 개발·배포·드리프트 모니터링)을 **LLM이 오케스트레이션하는 멀티에이전트 프레임워크**로 자동화. AutoML·MLOps·드리프트 인식 라이프사이클 최적화 포함.

> 🧠 [[웹-크롤링-기초]] 클라우드 루틴(수집)부터 배포·모니터링까지 한 파이프라인으로 묶는 청사진. Intelligence Entropy의 "드리프트 인식"과 직접 연결되는 라이프사이클 관점.

### 더존비즈온 — 일반 LLM의 약점을 친 '도메인 특화 전문 AI' 📄
*2026-06-16 · 택스워치*

"ChatGPT·클로드 같은 **일반 LLM은 세무사 시험 점수가 낮게** 나왔다"며, 더존비즈온은 세무사 수준의 전문 AI를 목표로 **도메인 특화 접근**을 추구. 세법~노무까지 '전문가의 시간'을 줄이는 것이 목표.

> 🧠 **[[Cogi-POC-Generator]] 핵심 통찰**: 범용 LLM을 그대로 쓰면 전문 도메인에서 정확도가 무너진다 → 내가 만드는 챗봇 시나리오 툴도 **도메인 지식(가이드·규칙) 주입 / 시나리오 제약**이 품질을 가른다는 근거. [[mailer|CS SmartHub]]의 CS 가이드를 도메인 지식원으로 붙이는 방향과 맞닿음. (위 LegalHalluLens "법률 AI 환각 ~52%"와 같은 메시지)

### 포스코DX — '1인 N에이전트' 멀티에이전트 업무 자동화 📄
*2026-06-18 · 헤럴드경제 / 이데일리*

포스코DX가 **멀티 에이전트 기반 업무 자동화**로 전사 AX 혁신 가속. 'AX 해커톤 2026'에서 적응형 멀티에이전트 커뮤니케이션 어시스턴트 등 6개팀 선정, 대상은 **'AI 기반 화면 설계 자동화 플랫폼'(PIXEL팀)**.

> 🧠 **업무 자동화 직결**: "한 사람이 여러 AI 에이전트를 부린다(1인 N에이전트)"는 그림은 [[내-프로필]]의 "인력 부족을 자동화 도구로 메운다"와 정확히 같은 발상. 'AI 화면 설계 자동화'는 [[Cogi-POC-Generator]]의 시나리오/화면 생성과 결이 같음.

### 한국 딜로이트 — 생성형 → 에이전틱 → 멀티에이전트, 거버넌스까지 통합 📄
*2026-06-17 · 한국경제*

AI 전담 조직·자체 플랫폼으로 전사 AI 전환. **생성형 AI → 에이전틱 AI → 피지컬 AI**로 적용 범위 확장, 진단·전략·로드맵부터 **거버넌스·리스크 관리, 멀티 에이전트 구조까지 통합**.

> 🧠 글로벌 섹션의 "에이전트 신뢰성/거버넌스" 흐름이 국내 컨설팅 실무로 내려온 사례. 단순 챗봇이 아니라 **거버넌스·리스크 설계가 멀티에이전트 도입의 필수 항목**이 됐다는 신호.

### 마키나락스 — LLM + RAG + XAI로 '자연어로 설명하는' 스마트정수장 📄
*2026-06-18 · AI타임스 / 한국수자원공사*

**LLM·RAG(검색증강생성)·XAI(설명가능 AI)**를 통합한 스마트 정수장 운영환경 구축. **AI 판단 결과를 자연어로 설명**하는 기능 포함.

> 🧠 RAG+XAI 조합은 "답을 주되 **왜 그렇게 판단했는지 설명**"하는 패턴 — CS 챗봇/[[Cogi-POC-Generator]]에서 답변 근거 표기와 직결. 위 ProvenanceGuard(출처 검증)·의사결정 인텔리전스(설명 가능성)와 같은 묶음.

### 정부 물가 조기경보 — LLM으로 가격정보 추출·표준화 자동화 📄
*2026-06-18 · 헤럴드경제 / 데일리안*

라면·빵 등 가공식품 13개+공산품에 **LLM으로 필요 정보를 추출·표준화**하는 가격 조기경보 시스템 구축. 별도로 생성형 AI '알뜰 소비 앱'(가격 지도·최적 구매처 추천)도 내년 하반기 시범.

> 🧠 **"비정형 데이터 → LLM 추출·표준화"**는 내가 [[웹-크롤링-기초]]로 수집한 자료를 정규화하는 작업과 같은 패턴. 정부도 LLM을 ETL 추출기로 쓴다는 실사용 레퍼런스.

### 그 외 국내 단신 📄
- **모노클 AI 플랫폼** (마스턴투자운용 임직원 교육) — **클로드·제미나이 등 여러 LLM을 한 환경에서** 활용. 🧠 [[내-MCP-커넥터-환경]]처럼 멀티 LLM을 한 곳에 모으는 사내 도입 패턴.
- **모비젠 Graphio** — **온톨로지 기반 데이터 플랫폼** + 공통/멀티 에이전트 협업. 🧠 기관 고유 용어·규칙 반영 = 더존비즈온 도메인 특화와 같은 메시지.
- **디자이노블 '스톰'** (파리 비바테크) — 멀티모달 온톨로지로 패션·제조 데이터를 AI 지식화, AI 에이전트 설계·운영 도구.
- **HPE·베이넥스 → 스트라드비젼** — GPU 통합관리·학습/배포 파이프라인 자동화로 **GPU 활용률 ~70% 개선** (MLOps 자동화).
- **키로(Kiro)** — 환각을 줄이려 **'신경기호 AI'** 도입 + **업무에 맞춰 더 작고 저렴한 모델을 골라 쓰는 전략**. 🧠 [[Cogi-POC-Generator]]에 LLM 붙일 때 "최신·최대 모델 ≠ 최적"이라는 비용·성능 균형 근거.

### 📌 흐름 메모: 비용·하드웨어·지정학 (저관련, 요약만) 🧠
- **모델 비용 경쟁**: 중국 오픈소스 LLM **'V4 프로'**가 오픈AI 상용 모델 대비 입력 비용을 크게 낮춤 → 국산 AI 글로벌 확장 위축 우려.
- **하드웨어**: SK하이닉스 **HBM4E 12단** 샘플(열저항 17%↓), 엔비디아는 **클라우드 밖 로컬 추론·에이전트 호스팅** 도전.
- **소버린/지정학**: 미 상무부의 **앤트로픽 모델 외국인 접근 제한**이 소버린 AI(NPU) 논의에 영향, 중국군 LLM(딥시크) 군사 도입. → 위 서울 오피스 개소의 "수출통제 논란"과 같은 맥락.

> ⚠ **취재수첩 — "직원이 일한 건가, AI가 일한 건가"** 📄 (디지털데일리): 한 사람이 여러 AI 도구로 업무를 굴리면서 **근무시간만으로 산출량을 가늠하기 어려워졌다**는 문제 제기. 🧠 자동화로 생산성을 올리는 나 같은 입장에선 "성과를 어떻게 보이게 만들 것인가"가 새 과제 — 자동화 결과물([[mailer|CS SmartHub]] 등)을 **명시적 산출물로 남기는 것**의 중요성.

## 의외의 연결점

> 🧠 이번 주 핵심 패턴: **"에이전트 신뢰성 3층 구조"** — ① 배포 전 시뮬레이션(Deployment Simulation) → ② 런타임 출처 검증(ProvenanceGuard) → ③ 장기 드리프트 방지(Intelligence Entropy). [[Cogi-POC-Generator]] 챗봇을 배포할 때 거쳐야 할 단계들.
> 🧠 **"평가의 신뢰성" 축이 새로 부각**: In-Context 평가 인식(모델이 테스트를 눈치챔) + LegalHalluLens(평균이 가리는 오류 분포)가 한목소리로 말한다 — **"평균 점수를 믿지 말고, 실사용과 닮은 입력으로·오류 유형을 분리해 측정하라."** Deployment Simulation이 왜 가치 있는지의 이론적 뒷받침.
> 🧠 **"제안-검증-종합" 구조가 정설로**: AI 에이전트 네트워크 신뢰성(2606.18121)이 통신이론으로, LegalHalluLens가 토론으로, ProvenanceGuard가 출처로 — 모두 **"단일 호출 대신 검증 노드를 끼운다"**는 같은 처방. 멀티에이전트 설계의 공통 골격.
> 🧠 **자동화 라이프사이클 한 줄로**: [[웹-크롤링-기초]](수집) → Trustworthy BDaaS(정제·배포·모니터링) → DRFLOW(다음 워크플로우 예측). [[내-프로필]]의 "반복 업무 자동화" 지향과 직접 맞물림.
> 🧠 **글로벌 연구 ↔ 국내 실무가 같은 곳을 가리킨다**: LegalHalluLens(법률 AI 환각 ~52%) ↔ 더존비즈온(일반 LLM 세무 점수 낮음) = **"범용 LLM은 전문 도메인에서 무너진다 → 도메인 특화·지식 주입이 답"**. ProvenanceGuard(출처 검증) ↔ 마키나락스 XAI·의사결정 인텔리전스 = **"답에 근거·설명을 붙여라"**. 제안-검증-종합 구조 ↔ 포스코DX 1인 N에이전트·딜로이트 멀티에이전트 거버넌스 = **"여러 에이전트 + 검증/거버넌스"**. 연구실의 결론이 국내 현장에 이미 적용되는 중.
