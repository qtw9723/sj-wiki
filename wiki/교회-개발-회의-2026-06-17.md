> 🧠 이 회의에서 **기존 설계 문서(backend-decisions.md)의 핵심 결정 A1이 번복**됐다. "나무는 절대 회수 안 한다(단조 증가)"에서 "회수한다(마지막 획득 나무부터)"로. 기술적으론 당연한 UX지만 이미 문서화·구현된 규칙을 뒤집는 일이 생겼다 — **설계 문서는 "일단 확정"이 아니라 회의마다 갱신 가능한 살아있는 문서**여야 한다는 교훈.

### 백엔드 ⚠️ 설계 변경 포함

| 결정 | 이전 vs 변경 |
|------|-------------|
| **성경 체크: 권 단위 bulk replace** — 해당 권 기존 데이터 전체 삭제 후 새 리스트 등록 | 이전: 단건 체크/해제 PATCH |
| **나무 회수 허용** — 재계산 후 기준 이하면 마지막 획득 나무부터 삭제 | ⚠️ 이전: 단조 증가·회수 안 함 → **번복** |
| 회수 시 배치된 나무도 삭제 가능 | 이전: 배치 영구 불변 → **수정** |
| **챌린지 기간: SQL INSERT로 직접 등록** | 어드민 UI 불필요 |
| 기간 이후 조회는 가능, **체크(쓰기)만 차단** | 기존과 방향 동일, 조회 허용 명시 |
| **나무 z-index: DB 시퀀스로 순차 증가** | 🆕 신규 요구사항 |
