# sj-wiki — 운영 스키마 (Schema)

이 저장소는 **sangjun(sjpark@mindwareworks.com)의 개인 지식·노하우·업무 기록 나무위키**다.
[Karpathy의 LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) 아이디어를 기반으로 한다.

핵심 원칙: **사용자는 원본 자료를 넣고 질문만 한다. Codex는 자료를 읽고 분석·연결·정리·유지보수하는 모든 작업을 담당한다.** 지식은 매번 RAG로 재추출하지 않고, 한 번 정리해 위키로 축적하고 항상 최신으로 유지한다.

## 1. 폴더 구조 & 소유권

```
sj-wiki/
├── AGENTS.md          # 이 운영 스키마 (규칙 정의)
├── raw/               # 원본 자료. 사용자가 넣음. Codex는 읽기만(수정·삭제 금지)
│   └── me/            # 사용자 본인 정보 원본만 따로 보관
└── wiki/              # 정리된 지식. Codex가 작성·소유·유지보수
    ├── index.md       # 전체 목차 (카테고리별 분류)
    ├── log.md         # 작업 이력 (append-only)
    ├── 내-프로필.md     # 사용자 정보 정리본 (모든 답변·takeaway의 관점 기준)
    └── *.md           # 개별 위키 페이지 (하위 폴더 없이 플랫)
```

**소유권 규칙**
- `raw/` : 사용자 영역. Codex는 **절대 수정·삭제하지 않고 읽기만** 한다. 디렉토리 구조 없이 **플랫**하게 관리(단, 본인 정보는 `raw/me/`에만).
- `wiki/` : Codex 영역. **하위 폴더 없이 플랫**하게 관리. 분류는 폴더가 아니라 `index.md`의 카테고리로 한다.

**⚠️ 회사 기밀 분리 원칙 (필수)**
- 이 vault(`sj-wiki`)는 **클라우드 동기화 대상 = 개인/포트폴리오 전용**이다.
- **회사(사내 AI 챗봇/MWW) 기밀은 절대 이 vault에 두지 않는다**: 사내 프로젝트 내부 설계(API/DB/시나리오), 회의록, 동료·임원 실명, 사내 NLP 플랫폼 내부 용어, 금액/인건비, 사내 경로 등.
- 회사 기밀은 **PC 전용 `/Users/sangjun/sj-wiki-work` vault**에만 보관한다.
- 자료를 정리하다 회사 기밀이 섞이면, 이 vault엔 일반화/제거하고 상세는 work vault로 보낸다. (`.gitignore`가 회사 패턴을 안전망으로 차단)

## 2. 운영 방법 (4가지)

### 운영 A. 자료 넣기 (Ingest)
사용자가 `raw/`에 자료를 넣거나 대화로 전달하면:
1. 원본이 대화로만 왔다면 `raw/`에 적절한 이름의 파일로 보관한다(본인 정보는 `raw/me/`).
2. 자료를 읽고 핵심을 분석한다.
3. `wiki/`에 정리된 페이지를 작성한다(아래 **3. 페이지 작성 규칙** 준수).
4. 관련된 기존 페이지에 `[[위키링크]]` 교차참조를 추가하고, 필요하면 기존 페이지도 갱신한다.
5. `wiki/index.md`의 카테고리에 새 페이지를 등록한다.
6. `wiki/log.md`에 작업 이력을 추가한다.

### 운영 B. 질문하기 (Query)
사용자가 질문하면:
1. **먼저 `wiki/`를 검색**해 관련 페이지를 찾아 그것을 근거로 답한다.
2. 항상 [[내-프로필]]의 관점에서, 사용자에게 의미 있는 답을 우선한다.
3. 답변 중 새로 가치 있는 결론이 나오면 위키 페이지로 정리해 다시 보관한다(필요 시).
4. 위키에 근거가 없으면 그렇다고 명시하고, 일반 지식으로 답한 뒤 위키화할지 제안한다.

### 운영 C. 건강검진 (Lint)
사용자가 "건강검진" 또는 점검을 요청하면 `wiki/` 전체를 점검한다:
- **모순/충돌**: 페이지 간 상충하는 주장
- **오래된 정보(stale)**: 갱신이 필요한 도구 버전·동향
- **고아 페이지(orphan)**: 어떤 페이지에서도 링크되지 않은 페이지
- **끊긴 링크**: 대상이 없는 `[[위키링크]]`
- **누락된 교차참조**: 연결되어야 하는데 안 된 페이지
- **분류 정합성**: `index.md`에 빠졌거나 잘못 분류된 페이지
점검 결과를 보고하고, 승인받아 수정한 뒤 `log.md`에 기록한다.

### 운영 D. 프로젝트 업데이트 (Project Update)
사용자가 "프로젝트 업데이트"(또는 특정 프로젝트명 + 업데이트)를 요청하면, 진행 중인 실제 프로젝트의 최신 상태를 위키에 반영한다:
1. 대상 프로젝트의 **라이브 저장소를 직접 확인**한다(`/Users/sangjun/IdeaProjects/<project>`): `git log`(마지막 업데이트 이후 커밋), 디렉토리/페이지 구조, README·AGENTS.md·docs 변화.
2. 파악한 진행사항으로 해당 위키 페이지의 **"진행 현황"·"진행사항 업데이트 로그"를 보강**(새 페이지 생성 X, 기존 보강).
3. 📄(저장소에서 확인)/🧠(추론) 구분을 유지하고, `log.md`에 기록한다.
- **등록된 진행형/포트폴리오 프로젝트** (frontmatter `flagship: true`):
  - [[mailer|CS SmartHub]] — 사용자가 직접 기획, 실사용자(사내 CS팀) 요구사항 기반, 지속 개발. 기본 업데이트 대상.
- **포트폴리오 프로젝트**는 그대로 포트폴리오로 쓸 수 있는 구조(문제·내 역할·기능·기술·진행현황·임팩트·회고)를 항상 유지한다.

## 3. 페이지 작성 규칙 (wiki/*.md)

**모든 위키 페이지는 다음 형식을 따른다.**

1. **YAML frontmatter**
   ```yaml
   ---
   title: 페이지 제목
   category: 카테고리명          # index.md 카테고리와 일치
   tags: [태그1, 태그2]
   source: raw/파일명 또는 URL    # 출처(있으면)
   created: YYYY-MM-DD
   updated: YYYY-MM-DD
   ---
   ```

2. **맨 앞 callout — 사용자 관점의 핵심 takeaway (필수)**
   본문 시작 전에 **반드시** 아래 callout을 넣는다. [[내-프로필]]을 고려해 "이 내용에서 sangjun이 가져가야 할 핵심"을 첫 문단에서 바로 확인할 수 있게 작성한다.
   ```markdown
   > [!tip] 핵심 takeaway
   > (사용자의 커리어·관심사·현재 업무 관점에서 본 1~3줄 요약.
   >  예: "이건 네가 만드는 챗봇 시나리오 툴에 바로 적용 가능", "업무 자동화 관점에서 이 부분만 보면 됨" 등)
   ```

3. **본문 — 상세 내용**
   callout 아래에 관련 상세 내용을 마크다운으로 정리한다.

4. **교차참조 — `[[위키링크]]` 적극 활용 (필수)**
   - 본문에서 다른 페이지·개념·도구를 언급하면 `[[위키링크]]`로 연결한다.
   - 아직 없는 페이지라도 `[[이름]]`으로 링크해두면 향후 작성 후보가 된다.
   - 페이지 하단에 `## 관련 문서` 섹션으로 연결을 모아둔다.

**파일명**: 한글 가능, 공백 대신 `-` 사용 권장(예: `LLM-챗봇-시나리오-툴.md`).

5. **이미지/스크린샷**
   - 원본 이미지는 `raw/assets/`에 둔다(플랫). 파일명: `<프로젝트>_<무엇>[_YYYYMMDD].확장자` (공백 X, 소문자·하이픈; 전 vault에서 유일하게).
   - 위키 페이지에서 `![[파일명.png]]`로 임베드하고 바로 아래 *이탤릭 1줄 캡션*으로 무슨 화면인지 설명한다. 가능하면 기능/주제별로 묶는다.
   - 이미지는 **자동 위키화(launchd) 대상에서 제외**됨 → 사용자가 "스크린샷 정리해줘" 등으로 요청할 때 해당 페이지에 임베드한다.

## 4. index.md 규칙
- 전체 페이지를 **카테고리별**로 나눠 `[[위키링크]]` 목록으로 관리한다.
- 기본 카테고리(필요 시 추가·조정):
  - **나에 대하여** (프로필·커리어·스킬)
  - **AI/업계 동향** (기사·트렌드)
  - **도구/스킬** (활용 중·희망 도구)
  - **프로젝트** (작업한 코드·개선점)
  - **회의록**
  - **개념/지식** (일반 정리)
- 자료를 넣을 때마다 갱신한다.

## 5. log.md 규칙
- append-only. 최신 항목을 위로 추가한다.
- 형식: `## [YYYY-MM-DD] 운영유형 | 제목` (운영유형: 자료넣기 / 질문 / 건강검진 / 셋업)
- 그 아래 한두 줄로 무엇을 했고 어떤 페이지가 영향받았는지 기록한다.

## 6. 프로필 참조 규칙
- [[내-프로필]]은 모든 답변과 takeaway의 **관점 기준**이다.
- 사용자가 본인에 대한 새 정보를 주면 `raw/me/`에 보관하고 [[내-프로필]]을 갱신한다.

## 7. 출처 표기 규칙 (Provenance) — 채팅·위키 모두 강제
**모든 주장은 "자료에서 찾은 것"인지 "Codex가 판단한 것"인지 구분해 밝힌다.** 사용자가 둘을 항상 분간할 수 있어야 한다.

- **분류 2종**
  - 📄 **자료** = raw 원본(또는 기존 위키 페이지)에 **명시된** 사실. 가능하면 `파일명`·`Case/Step·줄` 등 위치까지 댄다.
  - 🧠 **판단** = Codex의 추론·종합·제안·연결(의외의 연결점 포함). raw에 그대로 없는 모든 것.

- **채팅 답변에서**
  - 사실·요구사항·체크리스트 항목마다 📄/🧠를 표기한다. 특히 "확인 사항 / 요구사항 / 해야 할 일" 목록을 줄 때, 각 항목이 **자료에 있던 것(📄+출처)**인지 **Codex가 필요하다고 본 것(🧠)**인지 반드시 구분한다.
  - 📄와 🧠를 **한 항목에 뭉뚱그리지 않는다**(예: 자료의 개발 흐름을 "시연 확인 사항"처럼 섞지 않기).
  - raw에 근거가 없으면 "raw에 없음 / 불명"이라고 **명시**하고 단정하지 않는다.

- **위키 페이지에서**
  - frontmatter `source:`를 항상 채운다.
  - 본문에서 추론·제안·연결은 🧠 또는 "추론:" 으로 표기한다. `## 의외의 연결점`·핵심 takeaway·`## 제안`은 **판단 영역**임을 전제로 한다(자료 사실과 시각적으로 분리).
  - 자료끼리 충돌하거나 자료가 추론과 어긋나면 숨기지 말고 ⚠로 드러낸다.

- **확정/정정 추적**: 사용자가 구두로 확정·정정한 내용(📄로 승격)은 해당 페이지에 `확정 사항(날짜, 사용자 확인)` 노트로 남기고, 어긋나는 원본 자료는 "폐기/구버전"으로 표시한다.

## 8. 작업 전 pull · 변경 후 push (필수)
- 이 vault는 **private git에 연결**돼 있다(`origin/main`, GitHub `qtw9723/sj-wiki`). 여러 기기에서 동기화하므로 아래 순서를 지킨다.
- **① 작업 전 pull**: 위키화·편집을 시작하기 전에 **반드시 `git pull --rebase origin main`** 으로 원격 최신을 먼저 반영한다(다른 기기 변경과의 충돌 예방).
- **② 변경 후 push**: `wiki/`·`raw/`를 변경했으면 작업 마무리에 **반드시 커밋+푸시**한다: `git add -A && git commit -m "…" && git push`. ("위키화가 끝나면 반드시 push" — 사용자 요구.)
- 커밋 전 `git status`로 회사 기밀/시크릿이 staged되지 않았는지 확인한다(`.gitignore` 안전망 있음).
- 자동 위키화(launchd `.wiki-sync.sh`)는 **시작 시 pull --rebase → 위키화 성공 시 commit → pull --rebase → push**까지 자동 수행(실패 시 로그에 경고).

## 9. 다른 기기에서 시작하기 (Bootstrap)
이 vault를 새 기기에서 쓸 때(이 섹션도 git으로 함께 동기화됨):
1. **클론**: `git clone https://github.com/qtw9723/sj-wiki.git`
2. **Obsidian**: 폴더를 vault로 열어 보기·편집(단, Obsidian은 이 AGENTS.md 규칙을 *실행하지 않음* — 단순 노트로 표시).
3. **Codex**: **클론한 폴더를 작업 디렉토리로 Codex를 실행**하면 이 AGENTS.md가 프로젝트 지침으로 자동 로드되어 하네스가 작동한다. 위키 운영(자료 넣기·질문·건강검진·프로젝트 업데이트)은 Codex로 한다.
4. **git 규약(=§8)**: 작업 전 `git pull --rebase origin main`, 변경 후 `git add -A && git commit && git push`.
5. **메모리 주의**: `~/.Codex/.../memory`의 개인 메모는 이 repo에 포함되지 않는다(기기 종속). 다만 **운영 규칙 전부가 이 AGENTS.md에 자급자족형으로** 있어 규칙 적용에는 문제없다. 새 기기에서 중요한 사실을 기억시키려면 그 기기에서 별도로 메모리에 저장.
6. **자동 위키화(선택)**: 이 기기에서도 매일 자동 위키화를 원할 때만 launchd 재설정.
   - `.wiki-sync.sh`의 `/Users/sangjun/...` 경로·PATH·Codex 바이너리 위치를 그 기기에 맞게 수정.
   - `~/Library/LaunchAgents/com.<user>.sj-wiki-sync.plist`(09:00·18:00)를 만들고 `launchctl bootstrap gui/<UID> <plist>`로 등록.
   - 헤드리스 Codex 인증은 로그인 세션 키체인 접근이 필요(맥 launchd는 가능). 비-GUI 환경은 `Codex setup-token`.
   - ⚠ 자동 위키화·자동 push는 **주 기기 1대에서만** 돌리는 걸 권장(여러 대 동시 push 충돌 방지). 나머지 기기는 수동 pull/push.
7. **기밀 분리 재확인(=§1)**: 회사 기밀은 **PC 전용 `sj-wiki-work` vault**에만. 이 동기화 vault엔 절대 두지 않는다.
